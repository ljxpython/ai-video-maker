# AI Video Maker 开发规划：P1 到 P9

更新时间：2026-06-09

本文档记录 P0 主链之后的后续开发规划。当前 P0 已经打通：

```text
ai-video-maker
-> video-brief
-> video-plan
-> video-script
-> browser-capture 可选
-> voice-subtitle
-> edit-render
-> qa-revision
-> publish-package
```

后续目标不是继续堆 CLI，而是把项目打磨成“用户用 skill 对话，AI 自动判断下一步并执行内部 harness”的产品形态。

## 规划原则

- 用户入口永远是 `ai-video-maker` skill。
- CLI 和 harness 只作为内部执行工具。
- 每个阶段都必须有可检阅产物、handoff、测试和验收标准。
- `brief`、`plan`、`execution`、`upload`、`publish` gate 不能跳过。
- 涉及账号、上传、发布、桌面 GUI、副作用操作时，必须动作前确认。
- 每完成一个阶段，更新本文档和 `docs/TODO-功能实现路线图.md`。

## 方案确认状态

- P1 方案已确认：第一版只做 `next`，不做自动执行。
- P2 方案已确认：第一版做有限 Playwright action DSL，不做泛化 RPA，不处理登录态。
- P3 方案已确认：第一版只做 QA issue ledger + revision plan，不自动重跑。
- P4 方案已确认：第一版并入 `publish-package`，生成封面、章节、标题候选和 metadata QA，暂不拆独立 skill。
- P5 方案已确认：第一版只支持两个固定 profile：`youtube_16x9` 和 `shorts_9x16`，竖屏独立重排，不做简单拉伸。
- P6 方案已确认：第一版做“终端输出卡片化”，暂不接 `asciinema`，命令必须来自已确认计划或安全 allowlist。
- P7 方案已确认：第一版做 `$chrome` / `$computer-use` 的操作计划和结果记录适配器，不做账号侧全自动上传和发布。
- P8 方案已确认：第一版做 YouTube upload dry-run 和上传计划，真实上传后置到 `upload` gate，公开发布后置到 `publish` gate。
- P9 方案已确认：第一版做 skill 安装指南、manifest、安装脚本 dry-run 和校验工具，不做复杂插件市场。

## P1：Orchestrator next 状态机

### 最佳建议

第一版只做“建议下一步 + gate 阻断 + 用户可读说明”，不要做自动执行。

不要一上来实现：

```text
ai-video-maker continue 自动一路跑到底
```

原因：

- 容易绕过 hard gate。
- 容易在 `execution`、`upload`、`publish` 这类高风险阶段误执行。
- 当前最需要的是把“下一步该调用哪个 skill”变成稳定判断，而不是追求全自动。

第一版只实现：

```bash
ai-video-maker next --run runs/<run_id>
```

`next` 负责告诉 AI 和用户：

- 当前阶段是什么。
- 用户需要检阅什么。
- 是否卡在 hard gate。
- 下一步建议调用哪个 skill。
- 如果失败，建议回哪个 skill 返修。

### 目标

让用户说“继续”或“确认”，系统能根据当前 run 状态和最近 handoff 自动判断下一步应该调用哪个 skill。

### 期望效果

用户不需要记住：

```text
video-brief -> video-plan -> video-script -> voice-subtitle -> edit-render ...
```

用户只需要说：

```text
确认 brief，继续
```

AI 应该能输出：

```text
已确认 brief。下一步建议调用 video-plan，我将生成 storyboard、asset plan 和 capability plan。
```

### 怎么开发

新增模块：

```text
src/ai_video_maker/orchestrator.py
```

核心职责：

- 读取 `state.json`、`approvals.yml`、最近 handoff。
- 找到当前阶段：
  - 没有 brief：推荐 `video-brief`
  - `brief` 未确认：停在 `brief` gate
  - 没有 plan：推荐 `video-plan`
  - `plan` 未确认：停在 `plan` gate
  - 有 GUI 能力且 `execution` 未确认：停在 `execution` gate
  - 有 handoff：使用 `next_skill_suggestion`
  - `publish-package` 完成：停在 `upload` gate
- 输出统一的下一步说明。

新增内部 CLI：

```bash
ai-video-maker next --run runs/<run_id>
```

可选新增：

```bash
ai-video-maker continue --run runs/<run_id>
```

`continue` 暂不开发。等 `next` 稳定后再讨论是否支持软 review 阶段自动推进。硬 gate 永远必须使用明确确认。

### 产物

```text
runs/<run_id>/orchestrator/next.md
runs/<run_id>/orchestrator/next.yml
```

`next.yml` 示例：

```yaml
status: ready
current_stage: voice-subtitle
next_skill_suggestion: edit-render
next_gate: null
user_action_required: false
message: "语音和字幕已生成。请检阅后继续 edit-render。"
```

### 测试与验收

单元测试：

- 没有 brief 时推荐 `video-brief`。
- brief 未确认时返回 `next_gate: brief`。
- plan 未确认时返回 `next_gate: plan`。
- execution 未确认且需要 browser 时返回 `next_gate: execution`。
- `voice-subtitle` handoff 后推荐 `edit-render`。
- `qa-revision` 失败时推荐对应 `revision_skill_suggestion`。
- `publish-package` 后停在 `upload` gate。
- 任何 hard gate 未确认时，`next_skill_suggestion` 不能让系统继续执行后续副作用步骤。

集成验证：

```bash
".venv/bin/python" -m unittest discover -s "tests"
".venv/bin/ai-video-maker" next --run "runs/<run_id>"
```

验收标准：

- 用户只说“继续”，软 review 阶段可自动建议下一步。
- 用户没有确认 hard gate 时，系统不能继续执行有风险步骤。
- 输出文案能直接展示给用户，不要求用户读 CLI 输出。

## P2：browser-capture 结构化网页操作

### 最佳建议

第一版做“小而稳”的 Playwright action DSL，不做泛化 RPA，不处理登录态，不接 YouTube Studio。

推荐边界：

- 输入来自 `script/screen_actions.yml`。
- 执行器只支持有限动作集合。
- 每一步必须有 `id`，必须能记录 PASS/FAIL。
- 每一步尽量保存截图或证据。
- 失败时立即停止，输出 `needs_revision` handoff。
- 登录态、账号后台、YouTube Studio 全部留给 P7 `$chrome`。

不要第一版就做：

- 自然语言任意网页操作。
- 自动寻找元素。
- 自动处理登录、验证码、支付、上传。
- 复杂分支和循环。

原因：

- 技术视频录制追求可复现，不是做通用浏览器机器人。
- DSL 越小，越容易测试，也越容易从 `video-script` 自动生成。
- 对用户来说，稳定录出一段可用演示，比偶尔成功的“全自动网页 AI”更有价值。

### 目标

让 `browser-capture` 不只是打开网页等几秒，而是能按脚本执行网页演示动作。

### 期望效果

产品演示视频可以录制真实操作，例如：

```text
打开页面 -> 点击按钮 -> 填输入框 -> 滚动 -> 展示结果 -> 截图 -> 录制片段
```

### 怎么开发

#### 1. 定义 action DSL schema

扩展 `video-script`，新增结构化台本：

```text
runs/<run_id>/script/screen_actions.yml
```

示例：

```yaml
version: 1
target_url: "http://localhost:8000"
viewport:
  width: 1920
  height: 1080
recording:
  enabled: true
  output: "assets/browser/demo.webm"
actions:
  - id: open_home
    type: goto
    url: "http://localhost:8000"
  - id: click_demo
    type: click
    selector: "[data-testid='start-demo']"
  - id: fill_query
    type: fill
    selector: "input[name='query']"
    value: "AI Video Maker"
  - id: wait_result
    type: wait_for_selector
    selector: "[data-testid='result']"
  - id: screenshot_result
    type: screenshot
    output: "assets/browser/result.png"
```

第一版字段：

| 字段 | 要求 |
|---|---|
| `version` | 必填，当前为 `1` |
| `target_url` | 必填，必须是本地或公开网页 |
| `viewport` | 可选，默认 1920x1080 |
| `recording.enabled` | 可选，默认 true |
| `actions[].id` | 必填，唯一 |
| `actions[].type` | 必填，必须属于有限动作集合 |

#### 2. 新增 action parser

新增模块：

```text
src/ai_video_maker/browser_actions.py
```

职责：

- 读取 `script/screen_actions.yml`。
- 校验 action schema。
- 规范化默认值。
- 输出可执行 action list。
- 给 `video-script` 和 `browser-capture` 共用。

#### 3. 扩展 video-script 输出

`video-script` 继续保留：

```text
script/screen_actions.md
```

同时新增：

```text
script/screen_actions.yml
```

如果 pipeline 不需要 browser，则可以不生成 yml，或生成空 action list。

#### 4. 扩展 browser-capture 执行器

扩展 `browser_capture_workflow.py`：

- 读取 `screen_actions.yml`。
- 支持 action 类型：
  - `goto`
  - `click`
  - `fill`
  - `press`
  - `scroll`
  - `wait_for_selector`
  - `wait`
  - `screenshot`
  - `record_segment`
- 每步记录结果。
- 失败时生成可读报告，handoff 状态为 `needs_revision`。
- 没有 `screen_actions.yml` 时，保留当前 fallback：打开 `target_url`、截图、短录屏。

#### 5. 输出 step evidence

### 产物

```text
runs/<run_id>/script/screen_actions.yml
runs/<run_id>/assets/browser/demo.webm
runs/<run_id>/assets/browser/steps/*.png
runs/<run_id>/qa/browser_capture.md
runs/<run_id>/qa/browser_capture.json
runs/<run_id>/qa/browser_actions.json
runs/<run_id>/assets/browser/handoff.browser-capture.yml
```

`browser_actions.json` 示例：

```json
{
  "status": "passed",
  "actions": [
    {
      "id": "open_home",
      "type": "goto",
      "status": "passed",
      "screenshot": "assets/browser/steps/001_open_home.png"
    }
  ]
}
```

### 测试与验收

单元测试：

- action parser 校验非法 action。
- action parser 填充默认 viewport 和 recording。
- action id 重复时报错。
- `goto` 缺 url 报错。
- `click` / `fill` / `wait_for_selector` 缺 selector 报错。
- mock Playwright 验证每个 action 会调用对应方法。
- 缺 selector 时输出 `needs_revision`。
- 未确认 `execution` gate 时拒绝执行。
- 没有 `screen_actions.yml` 时仍可 fallback 到当前打开网页、截图、录屏流程。

真实 smoke：

```bash
cd examples/browser-local-demo
"../../.venv/bin/python" -m http.server 8000
```

再跑 `product_demo`，确认：

- 生成 `demo.webm`。
- 生成步骤截图。
- 报告列出每步 PASS/FAIL。

验收标准：

- 至少能完成一个 5 步网页演示。
- 失败时能明确指出失败 action。
- 每个成功 action 至少有日志记录，关键 action 有截图证据。
- 生成的视频能被 `edit-render` 混剪。
- 不处理登录态 Chrome，不触碰 YouTube Studio。

## P3：QA 返修闭环

### 最佳建议

第一版只做“结构化问题账本 + 返修计划”，不要直接自动重跑。

推荐边界：

- `qa-revision` 负责发现问题、分类问题、建议返修 skill。
- 新增 `qa/issues.yml` 作为问题账本。
- 新增 `revisions/revision_<id>.yml` 作为返修计划。
- Orchestrator P1 的 `next` 读取 `needs_revision` 和 issues，告诉用户该回哪个 skill。
- 用户确认返修后，再由对应 skill 执行重做。

不要第一版就做：

- 自动修改脚本。
- 自动重跑整条链路。
- 自动判断用户是否满意。
- 一个 QA 问题同时改多个阶段。

原因：

- 返修是最容易把项目搞乱的地方，必须先把问题归因和影响范围说清楚。
- 先有结构化 issue ledger，后续才能可靠地做局部重跑。
- 用户应该先知道“为什么返修、返修哪一步、会影响哪些产物”，再决定是否执行。

### 目标

让 `qa-revision` 不只是写报告，还能把问题结构化，并驱动后续局部返修。

### 期望效果

用户可以说：

```text
字幕遮挡按钮了，返修。
```

系统应输出：

```text
建议回到 edit-render，调整字幕位置后重渲染。后续需要重新跑 qa-revision 和 publish-package。
```

### 怎么开发

#### 1. 定义 QA issue schema

新增：

```text
runs/<run_id>/qa/issues.yml
runs/<run_id>/revisions/revision_<id>.yml
```

issue 示例：

```yaml
version: 1
issues:
  - id: subtitle_overlap
    status: open
    severity: medium
    category: subtitle
    source_skill: qa-revision
    revision_skill_suggestion: edit-render
    message: "字幕可能遮挡关键 UI"
    affected_outputs:
      - render/final_16x9.mp4
    evidence:
      - qa/screenshots/frame_6s.png
```

字段建议：

| 字段 | 说明 |
|---|---|
| `id` | 稳定问题 ID |
| `status` | `open`、`planned`、`resolved`、`ignored` |
| `severity` | `low`、`medium`、`high` |
| `category` | `audio`、`video`、`subtitle`、`browser`、`metadata`、`package` |
| `revision_skill_suggestion` | 建议返修 skill |
| `affected_outputs` | 受影响产物 |
| `evidence` | 截图、报告、ffprobe 等证据 |

#### 2. 定义 revision plan schema

`revision_<id>.yml` 示例：

```yaml
version: 1
revision_id: rev_subtitle_overlap_001
issue_ids:
  - subtitle_overlap
status: planned
revision_skill: edit-render
reason: "字幕遮挡关键 UI，需要调整字幕位置并重渲染。"
rerun_from: edit-render
rerun_after:
  - qa-revision
  - publish-package
blocked_by_gate: null
user_action_required: true
message: "建议回到 edit-render 调整字幕位置。确认后将重新生成视频、QA 和发布包。"
```

#### 3. 扩展 qa-revision

`qa-revision` 第一版需要把现有检查结果映射成 issue：

| 检查失败 | category | revision skill |
|---|---|---|
| 缺视频文件 | `video` | `edit-render` |
| 缺视频流 | `video` | `edit-render` |
| 缺音频流 | `audio` | `voice-subtitle` |
| 字幕为空 | `subtitle` | `voice-subtitle` |
| 关键帧截图失败 | `video` | `edit-render` |
| 发布 metadata 缺失 | `metadata` | `publish-package` |

#### 4. 新增 revise 命令，但只生成计划

新增内部命令：

```bash
ai-video-maker revise --run runs/<run_id> --issue subtitle_overlap
```

第一版 `revise` 不自动重跑，只生成 revision plan，并把状态交给 orchestrator。

后续等 P1 `next` 稳定后，再讨论：

```bash
ai-video-maker revise --run runs/<run_id> --issue subtitle_overlap --execute
```

`--execute` 也只能在 soft review 阶段执行，不能跳过 hard gate。

### 产物

```text
runs/<run_id>/qa/issues.yml
runs/<run_id>/revisions/*.yml
runs/<run_id>/revisions/*.md
```

同时更新：

```text
runs/<run_id>/qa/handoff.qa-revision.yml
```

handoff 在失败时应包含：

```yaml
status: needs_revision
issues:
  - subtitle_overlap
next_skill_suggestion: edit-render
revision_skill_suggestion: edit-render
user_action_required: false
```

### 测试与验收

单元测试：

- QA 失败生成 `issues.yml`。
- `audio_stream` 缺失路由到 `voice-subtitle`。
- `video_stream` 缺失路由到 `edit-render`。
- metadata 问题路由到 `publish-package`。
- `revise` 生成 revision plan，但默认不执行重跑。
- `issues.yml` 可以保留多个 open issue。
- 已忽略的 issue 不影响 `next` 建议。
- revision plan 能列出 `rerun_from` 和 `rerun_after`。

验收标准：

- 每个 QA 问题都能定位责任 skill。
- 返修不会跳过 gate。
- 用户能看到问题证据、影响范围和建议返修路径。
- 默认不自动重跑，确认后再执行。
- 后续执行返修时，必须生成新的 handoff 和 revision 记录。

## P4：发布包增强

### 最佳建议

第一版直接增强 `publish-package`，不要先拆 `thumbnail-package` 独立 skill。

推荐边界：

- `publish-package` 统一生成所有 YouTube 发布素材。
- 封面图先用可复现的 Pillow 生成，不调用外部图像模型。
- 章节从 storyboard 或字幕时间线生成。
- 标题候选、简介、标签先用模板和规则生成。
- metadata QA 只做静态检查，不接 YouTube API。
- `upload` 和 `publish` gate 必须继续保持 pending。

不要第一版就做：

- AI 生成复杂封面视觉。
- 自动上传封面或视频。
- 自动优化 YouTube SEO。
- 多平台 metadata 大而全。
- 独立 `thumbnail-package` skill。

原因：

- 当前发布包是主链末端，增强它能最快提升用户可用性。
- 封面和 metadata 先规则化，测试简单、产物稳定。
- 独立 skill 会增加 handoff 和返修复杂度，等 P4 稳定后再拆。

### 目标

让 `publish-package` 更像真实 YouTube 发布包，而不是只有基础标题、简介、标签。

### 期望效果

发布包包含：

```text
video.mp4
thumbnail.png
title.txt
title_options.md
description.md
chapters.txt
tags.txt
upload_checklist.md
```

### 怎么开发

#### 1. 固定发布包目录结构

第一版输出：

```text
package/
  video.mp4
  thumbnail.png
  title.txt
  title_options.md
  description.md
  chapters.txt
  tags.txt
  metadata_qa.yml
  upload_checklist.md
  handoff.publish-package.yml
```

#### 2. 封面图生成

扩展 `publish_package_workflow.py`：

- 优先读取 `qa/screenshots/frame_6s.png` 作为背景。
- 如果没有关键帧截图，则用纯色标题卡 fallback。
- 使用 Pillow 叠加：
  - 项目名或视频标题。
  - 一句副标题。
  - 简单角标，例如 `AI Video Maker`。
- 输出固定尺寸 `1280x720`。
- 不使用本地绝对路径或私人信息。

#### 3. 章节生成

章节来源优先级：

1. `plan/storyboard.yml` 的 sections。
2. `subtitles/captions.srt` 的时间线。
3. 默认章节 fallback。

输出：

```text
00:00 开场与目标
```

## P5：多格式渲染

### 最佳建议

第一版只支持两个固定 profile：

```text

shorts_9x16
```

横屏仍然是主版本，竖屏是附加版本。竖屏必须独立布局或裁切重排，不能把横屏视频简单压扁或拉伸。

推荐边界：

- `edit-render` 支持 `--profile` 和 `--profiles`。
- 默认仍输出 `render/final_16x9.mp4`，保持旧链路兼容。
- 竖屏输出 `render/final_9x16.mp4`。
- `publish-package` 按平台分目录：
  - `package/youtube/`
  - `package/shorts/`
- 先不做自动发布到 Shorts 或抖音。
- 先不做一堆平台 profile，只打磨两个最常用比例。

不要第一版就做：

- 任意分辨率自定义。
- 自动把横屏裁成所有平台。
- 横屏直接压扁成竖屏。
- 复杂动效重排。
- 多语言字幕和多音轨。

原因：

- 多格式最容易在画面构图和字幕可读性上翻车。
- 两个固定 profile 足够验证 renderer 架构是否能扩展。

## P6：terminal-capture 终端录制

### 最佳建议

第一版做“终端输出卡片化”，不要先接 `asciinema`。

推荐边界：

- `terminal-capture` 读取已确认的终端操作计划。
- 命令必须来自 `script/terminal_actions.yml`。
- 只执行安全 allowlist 内的命令。
- 每条命令记录 stdout、stderr、exit code、耗时和工作目录。
- 命令输出生成可混剪的终端卡片图片。
- `edit-render` 后续把终端卡片当作素材混剪。
- 失败命令不自动修复，只生成 QA 报告和 handoff。

不要第一版就做：

- 执行用户临时随口给出的任意命令。
- 执行删除、覆盖、系统配置、密钥读取、全局安装等高风险命令。
- 接 `asciinema` 并转视频。
- 自动根据报错改代码。
- 在没有 plan / execution gate 的情况下运行命令。

原因：

- 终端演示的核心价值是把安装、运行、测试过程讲清楚，不是先追求“像真人敲命令”。
- 文本卡片稳定、可测试、便于混剪，也更容易规避敏感路径和密钥输出。
- `asciinema` 后续可以作为增强项，但第一版先把命令安全和素材产物打牢。

### 目标

支持仓库讲解和技术教程里的终端安装、运行、测试过程。

### 期望效果

仓库讲解可以展示：

## P7：Chrome / computer-use 能力适配器

### 最佳建议

第一版做“操作计划 + 结果记录 adapter”，不要做完整 GUI 自动化平台。

推荐边界：

- `$chrome` 用于需要用户现有登录态的网页，例如 YouTube Studio、后台系统、私有产品页。
- `$computer-use` 用于桌面 app、文件选择器、原生系统对话框、剪辑软件。
- 第一版只生成操作计划，并记录工具辅助执行后的结果。
- 所有真实 GUI 操作前必须检查 `execution` gate。
- 涉及上传必须检查 `upload` gate。
- 涉及公开发布必须检查 `publish` gate。
- 只记录用户可见的截图、页面标题、操作结果和人工备注。

不要第一版就做：

- 读取 Chrome cookie、localStorage、密码、token。
- 自动登录账号。
- 自动绕过验证码、二次验证或风控。
- 自动上传或发布 YouTube 视频。
- 自动操作生产系统里的破坏性按钮。
- 把 `$chrome` / `$computer-use` 做成无边界通用机器人。

原因：

- 登录态和桌面 GUI 是高风险区域，必须先把 gate 和证据链做清楚。
- 对视频制作来说，第一版真正需要的是“能把已登录页面和桌面操作纳入素材采集”，不是替用户偷偷做账号操作。
- YouTube 上传发布应该放到 P8，并且必须独立走 `upload` / `publish` gate。

### 目标

支持需要登录态 Chrome 和桌面 GUI 的演示，但保持安全 gate。

### 期望效果

- `$chrome`：用于 YouTube Studio、登录后台、私有产品页面。
- `$computer-use`：用于桌面软件、文件选择器、OBS、剪辑软件。
- 操作前给用户一份明确计划。
- 操作后生成截图、记录和 handoff，供 `edit-render` 混剪或供 `qa-revision` 检查。

```text
计划 GUI 操作 -> 用户确认 execution -> 工具辅助操作 -> 记录截图和结果
```

P7 不做：

```text
自动上传 YouTube
自动点击发布
自动修改账号设置
```

这些都放到 P8，并且必须在发布包完整、`upload` / `publish` gate 确认后再做。

### 产物

```text
runs/<run_id>/plan/chrome_operation.yml
runs/<run_id>/plan/desktop_operation.yml
runs/<run_id>/assets/chrome/*.png
runs/<run_id>/assets/desktop/*.png
runs/<run_id>/qa/chrome_operation.md
runs/<run_id>/qa/chrome_operation.json
runs/<run_id>/qa/desktop_operation.md
runs/<run_id>/qa/desktop_operation.json
runs/<run_id>/assets/chrome/handoff.chrome-capture.yml
runs/<run_id>/assets/desktop/handoff.desktop-capture.yml
```

handoff 示例：

```yaml
status: ready_for_next
source_skill: chrome-capture
next_skill_suggestion: edit-render
artifacts:
  screenshots:
    - assets/chrome/open_studio.png
  report: qa/chrome_operation.md
safety:
  sensitive_storage_accessed: false
  upload_performed: false
  publish_performed: false
```

### 测试与验收

单元测试：

- 未确认 `execution` gate 时拒绝。
- 涉及账号和上传时必须要求 `upload` gate。
- 涉及公开发布时必须要求 `publish` gate。
- adapter 不能声明或记录 cookie、localStorage、密码、token。
- 操作计划能列出 allowed_actions 和 forbidden_actions。
- 结果记录必须有 screenshot 或用户可见证据。
- upload / publish 动作在 P7 默认返回 blocked。
- 输出操作报告。

人工验收：

- 能打开目标页面或桌面 app。
- 能记录截图或结果。
- 不读取敏感会话数据。
- 不自动上传、不自动发布。
- 出错时能生成 `needs_revision` 或 `blocked` handoff，并说明需要用户做什么。

## P8：YouTube 上传与发布

### 最佳建议

第一版只做“YouTube 上传计划 + dry-run”，不要一开始就真实上传。

推荐边界：

- `publish-package` 仍然只负责生成本地发布包。
- P8 新增独立 `youtube-upload` skill。
- 第一版读取发布包，生成上传计划和 dry-run 报告，不发网络请求。
- 真实上传只能在 `upload` gate 确认后执行。
- 上传默认可见性必须是 `private`。
- 改成 `unlisted` 或 `public` 必须在 `publish` gate 确认后执行。
- `$chrome` 操作 YouTube Studio 只作为 fallback，优先级低于 YouTube API dry-run。

不要第一版就做：

- 自动读取浏览器登录态。
- 自动点击 YouTube Studio 的上传或发布按钮。
- 自动公开发布。
- 在没有 metadata QA 的情况下上传。
- 在没有用户确认账号、标题、简介、标签、可见性的情况下上传。
- 把 OAuth token、client secret、刷新 token 写入仓库或 run 产物。

原因：

- 上传和发布是整个项目风险最高的外部副作用，必须分成 dry-run、upload、publish 三层。
- 真实上传失败成本高，先把计划、元数据校验和 gate 机制打牢。
- 用户最怕的是误公开发布，所以默认 private 是硬规则。

### 目标

在 `publish-package` 之后，支持经过 gate 的 YouTube 上传和发布。

### 期望效果

系统先生成发布包，用户确认：

```text
确认 upload gate，上传为 private。
```

上传完成后再确认：

```text
确认 publish gate，设置为 unlisted。
```

第一版如果不做真实上传，也应该能输出：

```text
上传计划已生成
metadata 检查通过
当前停在 upload gate
下一步：确认 upload gate 后再执行真实上传
```

### 怎么开发

#### 1. 新增 youtube-upload skill

```text
skills/youtube-upload/SKILL.md
```

skill 职责：

- 读取 `publish-package` handoff。
- 检查发布包完整性。
- 生成上传计划。
- 明确当前停在哪个 gate。
- dry-run 时不发网络请求。
- 真实上传前要求用户确认 `upload` gate。
- 公开或改可见性前要求用户确认 `publish` gate。

#### 2. 新增 workflow 和 adapter

新增：

```text
src/ai_video_maker/youtube_upload_workflow.py
src/ai_video_maker/adapters/youtube_api.py
```

`youtube_upload_workflow.py` 负责：

- 读取发布包。
- 读取 gate 状态。
- 校验 metadata。
- 生成 upload plan。
- 调用 adapter 的 dry-run 或真实上传接口。
- 写入报告和 handoff。

`youtube_api.py` 负责：

- 定义 YouTube API adapter 接口。
- 第一版只实现 dry-run adapter。
- 后续真实上传时再接 OAuth 和官方 API。

#### 3. 固定上传阶段

P8 分三层：

| 阶段 | 是否联网 | 必需 gate | 输出 |
|---|---:|---|---|
| `plan` | 否 | 无，或停在 `upload` gate | 上传计划 |
| `upload` | 是 | `upload` | private 视频草稿 |
| `publish` | 是 | `publish` | 可见性变更结果 |

CLI 建议：

```bash
rtk ".venv/bin/ai-video-maker" youtube-upload --run "runs/<run_id>" --dry-run
rtk ".venv/bin/ai-video-maker" youtube-upload --run "runs/<run_id>" --execute-upload
rtk ".venv/bin/ai-video-maker" youtube-upload --run "runs/<run_id>" --execute-publish
```

默认不传参数时等同于 `--dry-run`。

#### 4. 上传计划 schema

新增：

```text
runs/<run_id>/upload/youtube_upload_plan.yml
```

示例：

```yaml
version: 1
platform: youtube
mode: dry_run
requires_gate: upload
source_package: package/youtube
video: package/youtube/video.mp4
thumbnail: package/youtube/thumbnail.png
metadata:
  title: package/youtube/title.txt
  description: package/youtube/description.md
  tags: package/youtube/tags.txt
visibility:
  upload_default: private
  requested_publish_visibility: unlisted
safety:
  metadata_qa_required: true
  upload_gate_required: true
  publish_gate_required: true
  dry_run_network_requests: false
```

#### 5. OAuth 和密钥策略

第一版只写配置文档，不要求用户立刻配置真实 OAuth。

后续真实上传时：

- OAuth 凭据只从环境变量或用户指定的外部路径读取。
- 不把 token 写入 `runs/`。
- 不把 client secret 写入仓库。
- 不在 QA 报告里打印密钥。
- 如果凭据缺失，输出 `blocked` handoff，而不是失败重试。

#### 6. `$chrome` fallback

如果用户不想配置 YouTube API，可以走 P7 的 `$chrome` 操作计划：

```text
youtube-upload -> 生成 YouTube Studio 操作计划 -> chrome-capture 记录执行结果
```

但 fallback 仍然必须遵守：

- 上传前确认 `upload` gate。
- 发布前确认 `publish` gate。
- 不读取 cookie、localStorage、密码、token。
- 不自动点击发布按钮。

### 产物

```text
runs/<run_id>/upload/youtube_upload_plan.yml
runs/<run_id>/upload/youtube_dry_run.md
runs/<run_id>/upload/youtube_dry_run.json
runs/<run_id>/upload/oauth_setup.md
runs/<run_id>/upload/handoff.youtube-upload.yml
```

真实上传后新增：

```text
runs/<run_id>/upload/youtube_upload_result.yml
runs/<run_id>/upload/youtube_publish_result.yml
```

handoff 示例：

```yaml
status: waiting_for_gate
source_skill: youtube-upload
next_gate: upload
next_skill_suggestion: youtube-upload
message: "YouTube 上传计划已生成。确认 upload gate 后才能上传 private 视频。"
safety:
  dry_run: true
  network_requests_performed: false
  upload_performed: false
  publish_performed: false
```

### 两条实现路线

1. YouTube Data API。
2. `$chrome` 操作 YouTube Studio。

推荐先做 API dry-run：

- 读取发布包。
- 校验 metadata。
- 生成上传计划。
- 不执行外部请求。

再做真实上传：

- OAuth 配置文档。
- `upload` gate 后上传。
- 默认 private。
- `publish` gate 后才改可见性。

### 测试与验收

单元测试：

- 没有 `upload` gate 不能上传。
- 没有 `publish` gate 不能公开。
- metadata 缺失时拒绝上传。
- dry-run 不发网络请求。
- 默认可见性必须是 `private`。
- `--execute-upload` 缺 OAuth 配置时输出 `blocked`。
- `--execute-publish` 没有 upload result 时拒绝。
- token、client secret、cookie 不能出现在报告和 handoff。
- metadata QA 未通过时不能进入 upload。
- `$chrome` fallback 必须生成操作计划，不直接执行发布。

集成验证：

```bash
rtk ".venv/bin/ai-video-maker" youtube-upload --run "runs/<run_id>" --dry-run
rtk ".venv/bin/python" -m unittest discover -s "tests"
```

人工验收：

- 测试账号上传 private 视频。
- 检查标题、简介、标签。
- 不自动公开。
- 确认 publish gate 前，视频不能变成 unlisted 或 public。
- OAuth 凭据不会进入 Git、run 产物或报告。

## P9：Skill 安装与分发形态

### 最佳建议

第一版做“安装指南 + manifest + dry-run 安装脚本 + 校验工具”，不要做复杂插件市场。

推荐边界：

- 外部用户 clone 仓库后，可以把 skills 安装到自己的 Codex skills 目录。
- 安装脚本默认 dry-run，只展示会安装什么。
- 支持 `--copy` 和 `--link` 两种模式。
- 已存在同名 skill 时默认不覆盖。
- 安装后可以跑校验命令，确认 skill frontmatter、名称、入口文档都正常。
- 文档必须只使用通用路径，不暴露维护者本地路径。

不要第一版就做：

- 自动修改用户 shell 配置。
- 自动覆盖用户已有 skills。
- 自动安装全局依赖。
- 发布到复杂 marketplace。
- 强制用户使用某一种 Codex 安装目录。

原因：

- P9 的目标是让项目可被别人用，不是做一个完整包管理器。
- 安装脚本一旦写错会污染用户本地 skill 目录，所以默认 dry-run。
- manifest 和校验工具先做起来，后续才能稳定支持更多 skills 和 agents。

### 目标

让外部用户更容易安装和使用这些 skills。

### 期望效果

用户 clone 仓库后，可以按文档把 skills 安装到 Codex skill 目录，或者由脚本自动链接。

安装完成后，用户能看到：

## 开发准入结论

结论：可以进入开发，但必须按低风险到高风险顺序推进，不建议先开发 P8 上传发布。

推荐开发顺序：

```text
P1 -> P2 -> P6 -> P3 -> P4 -> P5 -> P7 -> P8 -> P9
```

理由：

- P1 是 orchestrator，决定“用户说继续时下一步调用哪个 skill”。
- P2 和 P6 补齐网页演示和终端演示素材能力。
- P3 补齐返修闭环，否则后续越自动化越难定位问题。
- P4 和 P5 提升发布包和多格式成片质量。
- P7 才接入 `$chrome` / `$computer-use` 这类高风险 GUI 能力。
- P8 最后接真实 YouTube 上传发布，必须等 gate、metadata QA、P7 记录能力稳定。
- P9 放在功能稳定后做安装分发，避免把半成品 workflow 推给外部用户。

开发开工前的硬条件：

- 每个阶段都有明确产物和 handoff。
- `brief`、`plan`、`execution`、`upload`、`publish` gate 规则不能放松。
- 所有上传、发布、账号、桌面 GUI、副作用操作必须动作前确认。
- `runs/`、OAuth token、cookie、本地绝对路径、私有邮箱不能进入 Git。
- 每个 P 阶段完成后必须跑单元测试、目标 smoke、`git diff --check` 和隐私扫描。

如果 P8/P9 采用本文档边界，则当前方案已经足够进入开发阶段。建议第一个开发任务从 P1 `ai-video-maker next --run ...` 开始。
